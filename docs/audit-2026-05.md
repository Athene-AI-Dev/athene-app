### 1. `lib/telemetry/spans.ts` — OpenTelemetry Spans

**Purpose**: Provides span wrappers for vector search, LLM calls, tool calls, and SSE frames.

**Code Review**:

| # | Severity | Description | Status |
|---|----------|-------------|--------|
| 1 | Low | `context.with(ctx, () => fn(span))` — async context propagation unverified | ⚠️ Noted |
| 2 | Info | Lazy tracer init is correct pattern | ✅ OK |
| 3 | Info | `withVectorSearchSpan` now accepts `topK` parameter | ✅ OK |

**Detail on Issue #1**:
The `withSpan` function uses `context.with(ctx, callback)` to propagate the OTel context. In Node.js, OpenTelemetry uses `AsyncLocalStorage` for context propagation. When `callback` returns a Promise (as `fn(span)` does), the context SHOULD be available to all descendant async operations.

However, the `context.with()` implementation:
1. Saves current context
2. Sets `ctx` as active
3. Calls `callback()`
4. Restores previous context

If `callback()` returns a Promise, step 4 happens BEFORE the Promise settles. The question is whether `AsyncLocalStorage` properly propagates `ctx` to the Promise's `.then()` handlers.

**Verdict**: In practice, `AsyncLocalStorage` propagates the store to all async operations created within the scope where the store was set. So this SHOULD work. But it's untested.

**Production Readiness**: ✅ Ready (with note about untested async propagation)

### 2. `lib/auth/cached-clerk.ts` — Cached Clerk Auth

**Purpose**: Cache Clerk session validation to avoid per-request overhead.

**Bugs Found**:

| # | Severity | Description | Status |
|---|----------|-------------|--------|
| 1 | **Critical** | `match[2]` should be `match[1]` — regex only has one capture group | ✅ **FIXED** |
| 2 | Medium | Cookie extraction regex might not cover all Clerk cookie variants | ⚠️ Noted |
| 3 | Info | `cachedAuth` now properly integrated in `agent/route.ts` | ✅ OK |

**Bug #1 — Critical (FIXED)**:
```typescript
// BEFORE (broken):
const hash = crypto.createHash("sha256").update(match[2]).digest("hex").slice(0, 24);

// AFTER (fixed):
const hash = crypto.createHash("sha256").update(match[1]).digest("hex").slice(0, 24);
```

The regex `/(?:^|;)\s*(?:__session|__clerk[^=]*)=([^;]+)/` has ONE capture group: `([^;]+)`. This is `match[1]`. Using `match[2]` returns `undefined`, causing `crypto.createHash(...).update(undefined)` to throw.

**Bug #2 — Cookie regex coverage**:
The regex matches `__session` or `__clerk*` cookies. Clerk's actual cookie names include:
- `__session`
- `__clerk_db_jwt`
- `__clerk_org_id`
- `__clerk_user_id`

The regex `(?:__session|__clerk[^=]*)` matches all of these. ✅ OK.

**Integration fix**: `agent/route.ts` was updated to import and use `cachedAuth(req)` instead of bare `auth()` from `@clerk/nextjs/server`.

**Production Readiness**: ✅ Ready (after fix)

### 6. `tests/load/agent-stream.k6.js` — k6 Load Test

**Purpose**: Load test with 50 concurrent users measuring p50/p95/p99.

**Review**:

| # | Severity | Description | Status |
|---|----------|-------------|--------|
| 1 | Low | `firstTokenLatency.add(ttfbMs)` — k6 doesn't support streaming, so `ttfbMs` is total request time | ⚠️ Noted |
| 2 | Info | `getParams()` only sets `Authorization` header when token is present | ✅ OK |
| 3 | Info | SSE parsing handles both `data:` and `data: ` formats | ✅ OK |
| 4 | Info | Thresholds match actual metric names | ✅ OK |

**Issue #1 — First Token Latency**:
k6's `http.post()` waits for the full response body before returning. Therefore:
- `ttfbMs` = total time until k6 receives the COMPLETE response
- This is NOT the time to first byte or first token

The metric is Misleading. For true first-token latency measurement, use:
- Artillery with streaming support
- Custom Node.js script with `fetch()` + `ReadableStream`

**Workaround in code**: The comment documents this limitation. The `firstTokenLatency` metric is approximate.

**Production Readiness**: ⚠️ Partial — needs actual test run to validate thresholds

**To Run**:
```bash
npm run dev &  # start dev server
k6 run --vus 50 --duration 30s \
  -e BASE_URL=http://localhost:3000 \
  -e CLERK_TOKEN=<valid_jwt> \
  tests/load/agent-stream.k6.js
```

### 7. `supabase/migrations/20260506000000_owner_indexes.sql` — DB Indexes

**Purpose**: Add standalone indexes for `owner_user_id` on `documents` and `document_embeddings`.

**Review**:

| # | Severity | Description | Status |

| 1 | **Critical** (Fixed in prev round) | `CREATE INDEX CONCURRENTLY` inside transaction | ✅ **FIXED** |
| 2 | Info | Regular `CREATE INDEX` is fine for new/migration tables | ✅ OK |

**Note on `CONCURRENTLY`**:
- `CONCURRENTLY` cannot run inside a transaction
- Supabase runs migrations in a transaction by default
- Fix: Use regular `CREATE INDEX` in migration; apply `CONCURRENTLY` manually in production via SQL editor

**Production Readiness**: ✅ Ready (non-concurrent indexes are fine for dev/new tables)

For production tables with heavy traffic, apply concurrently outside transaction:
```sql
-- Run manually in Supabase SQL editor:
CREATE INDEX CONCURRENTLY idx_documents_owner_user_id ON documents(owner_user_id);
CREATE INDEX CONCURRENTLY idx_embeddings_owner_user_id ON document_embeddings(owner_user_id);
```

## Remaining Risks

1. **Load test not executed**: The k6 script is written but hasn't been run against a live instance. The target metrics (p95 < 300ms, < 1.5s, < 500ms) are unvalidated.

2. **OTel collector not configured**: The spans are created but there's no exporter/collector. Without `OTEL_EXPORTER_OTLP_ENDPOINT` or similar, spans go nowhere.

3. **Client-side SSE handling**: The route now emits `{ token: "..." }` frames. If the client doesn't handle this frame type, streaming will break.

4. **Redis dependency**: The caching layer falls back to no-op stubs if Redis env vars aren't set. For production, ensure `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` are set.

5. **`cachedAuth` session cookie extraction**: The regex for Clerk cookie names might not cover all variants. Monitor for cache misses.

6. **`context.with()` async propagation**: Unverified whether OTel context properly propagates to async continuations in `withSpan`. Consider testing with an actual OTel collector.

---

## Recommendations

1. **Run load tests** with a live app instance to validate the target metrics
2. **Set up OTel collector** (e.g., Jaeger, Honeycomb) to actually capture spans
3. **Add integration test** for `cachedAuth` to verify session cookie hashing works
4. **Monitor Redis hit rate** in production to validate cache effectiveness
5. **Ensure `X-Accel-Buffering: no` header** is set (already added) to prevent nginx from buffering SSE streams
6. **Test `withSpan` async propagation** by creating a child span inside an async function and verifying it's properly parented in the OTel collector
